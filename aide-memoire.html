1. Architecture générale

Frontend Angular 21

  • Port 4200 (ng serve)
  • Standalone components
  • Signals + Computed
  • Tailwind CSS v3
  • HttpClient → REST API

Backend PHP/Slim 4

  • Port 8000 (php -S)
  • Architecture MVC
  • Sessions PHP natives
  • Réponses JSON
  • PDO → MariaDB

Base de données MariaDB 10.6

  • Port 3307 (WampServer)
  • BDD : marketplace_artisanal
  • Tables : utilisateur, produit, commande…
  • Connexion via PDO (singleton)

Communication CORS + Session

  • withCredentials: true sur chaque requête
  • Cookie de session PHP transmis
  • CorsMiddleware autorise localhost:4200
  • Preflight OPTIONS géré côté Slim
Principe SPA : Angular charge une seule fois dans le navigateur. La navigation entre pages ne recharge pas le serveur — seules les requêtes API (GET/POST…) partent vers le backend PHP.
État réel du dépôt : le frontend Angular est maintenant compilé vers marketplace/public/app pour être servi par le front controller PHP en production ; le backend Slim expose les routes principales dans marketplace/app/config/routes.php ; les scripts SQL sont conservés dans marketplace/sql/.
Arborescence fonctionnelle du projet
marketplace/frontend/ = SPA Angular + routes + pages métier
marketplace/app/ = contrôleurs, modèles, middlewares, sécurité
marketplace/public/ = point d'entrée HTTP + assets build Angular
marketplace/sql/ = schéma, seed, scripts de migration
marketplace/test/ = vérifications et checklist API

4. Flux complet : d'un clic à un résultat

Exemple : l'utilisateur clique "Ajouter au panier" sur un produit

Diagramme réel de bout en bout
frontend/src/app/pages/product-detail/ → POST /api/statistiques-artisans → app/controllers/StatistiqueArtisanController.php → app/models/StatistiqueArtisanModel.php → MariaDB → frontend/src/app/pages/artisan/stats/
1
Clic dans le navigateur Le bouton Angular appelle (click)="addToCart(produit)" dans product-detail.component.ts
2
Service Angular cart.service.ts → méthode addItem(produit, quantite) → construit un FormData avec id_produit et quantite
3
Requête HTTP HttpClient.post('http://localhost:8000/api/cart', formData, { withCredentials: true }) — le cookie de session est envoyé automatiquement
4
CORS Middleware (PHP) CorsMiddleware.php reçoit la requête en premier. Si c'est un OPTIONS preflight → réponse immédiate 204. Sinon → ajoute les headers CORS et passe la main.
5
Session Middleware (PHP) SessionMiddleware.php → session_start() → $_SESSION est disponible avec l'ID utilisateur
6
Auth Middleware (PHP) AuthMiddleware.php → vérifie $_SESSION['user_id']. Si absent → retourne 401 Unauthorized. Sinon → passe la main.
7
Routeur Slim routes.php → mappe POST /api/cart vers CartController::add
8
Contrôleur PHP CartController.php → lit $_POST['id_produit'], appelle CartModel::add()
9
Modèle PHP → PDO → MariaDB CartModel exécute INSERT INTO ligne_panier ... via PDO sur marketplace_artisanal (port 3307)
10
Réponse JSON JsonResponder::json($response, ['success' => true, 'cart' => [...]]) → retourne le panier mis à jour
11
Mise à jour Angular cart.service.ts recharge le panier → le Signal items est mis à jour → le badge dans le header se rafraîchit automatiquement (réactivité Signal)
Même logique pour login : login.component.ts → auth.service.ts → POST /api/auth/login → AuthController::login → UtilisateurModel::findByEmail → password_verify() → $_SESSION['user_id'] = $user['id'] → JSON retourné → Signal currentUser mis à jour → navigation automatique vers /.

5. D'où partent les requêtes HTTP ?

Toutes les requêtes partent des services Angular (src/app/core/services/). Les composants ne font jamais de HTTP directement — ils appellent une méthode de service.
ServiceRequêtes émisesEndpoint PHP
auth.service.ts POST login, POST register, GET me, POST logout /api/auth/*
product.service.ts GET produits, GET produit/:id, POST, PUT, DELETE /api/produits/*
cart.service.ts GET panier, POST ajout, PUT quantité, DELETE item /api/cart/*
order.service.ts POST commande, GET historique, PATCH statut /api/commandes/*
artisan.service.ts GET artisans, GET artisan/:id, GET produits artisan /api/artisans/*
admin.service.ts GET stats globales, GET tous utilisateurs, PATCH rôle /api/admin/*
address.service.ts GET adresses user, POST nouvelle adresse /api/adresses/*

Comment Angular envoie les données

  • FormData pour tous les POST/PUT → PHP lit $_POST
  • withCredentials: true → le cookie de session PHP est inclus automatiquement
  • L'URL de base (http://localhost:8000) vient de environment.ts

6. En quoi Angular a aidé le développement ?

Architecture claire

  • Séparation stricte : composants / services / guards / modèles
  • Chaque fichier a une responsabilité unique
  • Facile à maintenir et à déboguer

Réactivité avec les Signals

  • signal() et computed() = état global réactif
  • Le badge du panier se met à jour tout seul
  • Le menu change selon le rôle sans code supplémentaire

Routage et guards

  • Navigation SPA sans rechargement de page
  • authGuard et roleGuard protègent les routes côté client
  • Lazy loading : le code admin n'est chargé que si on est admin

TypeScript + Interfaces

  • models.ts : contrats de données partagés
  • Le compilateur signale les erreurs de champs avant l'exécution
  • Autocomplétion dans l'IDE pour les champs PHP

Composants réutilisables

  • product-card, kpi-card, toast-container
  • Écrits une fois, utilisés partout
  • Tailwind CSS pour le style rapide et cohérent

HttpClient

  • Gestion propre des Observables (RxJS)
  • withCredentials centralisé dans les services
  • Typage fort des réponses API

7. En quoi Slim a aidé le développement ?

Routage expressif

  • $app->get('/api/produits', [...]) : une ligne = une route
  • Paramètres nommés : /api/produits/{id}
  • Regroupement logique des routes par domaine

Middleware en couches (PSR-15)

  • Session → CORS → Auth → RoleCheck → Controller
  • Chaque middleware est testable indépendamment
  • Ordre d'exécution garanti (pile LIFO)

Réponses PSR-7

  • Objet Response immuable et standard
  • JsonResponder centralise l'encodage JSON
  • Codes HTTP explicites (200, 201, 401, 403, 404…)

Légèreté

  • Aucun ORM imposé → PDO pur, requêtes optimisées
  • Pas de magie cachée contrairement à Laravel
  • Facile à comprendre pour un débutant

Séparation des responsabilités

  • Contrôleurs : logique métier uniquement
  • Modèles : accès BDD uniquement
  • Validators : validation des entrées séparée

Compatibilité

  • Fonctionne avec le serveur PHP built-in (pas besoin d'Apache)
  • BasePathResolver adapte le chemin automatiquement
  • Déploiement simple sur n'importe quel hébergeur PHP

8. Gestion des rôles et sécurité

Rôleid_roleGuard AngularMiddleware PHPAccès
Admin1 roleGuard(1) RoleMiddleware(1) Tout : artisans, produits, commandes, utilisateurs
Artisan2 roleGuard(2) RoleMiddleware(2) Ses produits, ses commandes reçues, ses statistiques
Client3 authGuard AuthMiddleware Panier, commandes passées, profil, adresses
Visiteur— Aucun guard Aucun middleware Catalogue, fiches produits, vitrines artisans
Double protection : les guards Angular empêchent l'accès aux pages côté client, mais le middleware PHP refuse aussi les requêtes API non autorisées. Un utilisateur qui tenterait d'appeler /api/admin/artisans sans être admin reçoit un 403 Forbidden du serveur, même s'il a contourné le guard.

9. Glossaire technique rapide

TermeDéfinition courte
SignalValeur réactive Angular 17+. Quand elle change, les composants qui l'utilisent se mettent à jour automatiquement.
computed()Signal dérivé : calculé automatiquement à partir d'autres signals. Ex : liste filtrée.
GuardFonction Angular qui s'exécute avant de charger une route. Retourne true (accès) ou redirige.
MiddlewareCouche PHP intercalée entre la requête et le contrôleur. S'exécutent dans l'ordre : Session → CORS → Auth → Role.
PSR-7Standard PHP pour les objets Request/Response. Slim, comme Laravel, l'implémente.
PDOPHP Data Objects : interface générique pour accéder aux bases de données (MySQL, MariaDB, SQLite…).
SPASingle Page Application. L'application charge une fois, la navigation ne recharge pas la page.
Lazy loadingLes modules Angular ne sont téléchargés qu'au moment où l'utilisateur en a besoin.
CORSMécanisme de sécurité navigateur. Le frontend (4200) et le backend (8000) sont sur des origines différentes → le serveur doit explicitement autoriser.
withCredentialsOption HttpClient qui inclut les cookies (session PHP) dans les requêtes cross-origin.
FormDataFormat de données POST. PHP le lit via $_POST. Utilisé à la place de JSON car PHP ne parse pas le body JSON nativement.
SingletonPattern de conception : une seule instance d'une classe. Ex : Database::getConnection() retourne toujours la même connexion PDO.
