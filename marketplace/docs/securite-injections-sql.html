Protection contre les injections SQL

Ce document explique simplement les protections mises en place pour empêcher qu’une donnée saisie par un utilisateur ne devienne une instruction SQL malveillante.

Le problème

Une injection SQL survient quand une valeur saisie par l’utilisateur est interprétée comme du code SQL par la base de données.

Le but des protections ici est de séparer clairement :

  • la requête SQL attendue
  • les valeurs transmises à cette requête

Ce qui a été mis en place

  • validation des champs attendus
  • whitelist des colonnes autorisées
  • requêtes préparées PDO
  • nettoyage des données entrantes
  • gestion d’erreurs contrôlée

Impact fonctionnel

Ces mécanismes couvrent aussi le flux de consultation produit, notamment la sauvegarde des statistiques artisan.

Chaque étape ajoute une barrière : on vérifie, on filtre, on nettoie, puis on exécute la requête avec des valeurs séparées du SQL.

Les protections principales

  1. Validation stricte : les champs obligatoires, numériques ou au bon format sont contrôlés avant écriture.
  2. Whitelist des champs : seuls les champs autorisés du modèle sont pris en compte.
  3. Requêtes préparées : la requête SQL et les valeurs sont envoyées séparément via PDO.
  4. Sanitize : les données sont nettoyées avant traitement ou enregistrement.
  5. Contrôle des colonnes dynamiques : une colonne non prévue est refusée lors d’un filtre ou d’une recherche.
  6. Gestion d’erreurs maîtrisée : les erreurs techniques ne sont pas exposées brutement à l’utilisateur.

Extraits de code

Validation — fichier concerné : app/models/validators/StatistiqueArtisanValidator.php

public function validate(array $data): ValidationResult
{
    $this->result = new ValidationResult();
    $this->validateRequired($data, ['date_consultation', 'ip_adress', 'id_produit', 'id_artisan']);
    $this->validateNumeric('id_artisan', $data);
    $this->validateNumeric('id_produit', $data);

    if (isset($data['ip_adress']) && filter_var($data['ip_adress'], FILTER_VALIDATE_IP) === false) {
        $this->result->addError('ip_adress', 'L adresse IP est invalide.');
    }

    return $this->result;
}

Whitelist / sanitize — fichier concerné : app/models/Model.php

protected static function sanitizeData(array $data): array
{
    return array_map(static fn($value) => static::sanitizeValue($value), $data);
}

public function fill(array $attributes): self
{
    $attributes = static::sanitizeData($attributes);
    foreach (static::$fields as $field) {
        if (array_key_exists($field, $attributes)) {
            $this->attributes[$field] = $attributes[$field];
        }
    }
    return $this;
}

Requête préparée — fichier concerné : app/models/Model.php

$stmt = static::getPDO()->prepare($sql);
foreach ($fields as $column => $value) {
    $stmt->bindValue(':' . $column, $value);
}
$stmt->execute();

try {

Résumé simple

Le principe de sécurité est très simple : on ne laisse jamais une valeur utilisateur dicter la structure d’une requête SQL.
