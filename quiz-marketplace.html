A. Architecture générale

1 Sur quel port tourne le serveur PHP/Slim 4 du projet marketplace ?
Réponse : C. Le serveur est lancé avec php -S localhost:8000 -t public. Le fichier public/index.php reçoit toutes les requêtes sur ce port. Pourquoi pas 80 ? Apache (WampServer) ne pouvait pas démarrer, donc on utilise le serveur intégré PHP directement.
2 Sur quel port tourne MariaDB dans ce projet ?
Réponse : A. Dans app/config/databaseConfig.php : 'port' => 3307. Pourquoi pas 3306 ? MySQL 5.7 (port 3306) utilise l'auth GSSAPI pour root, ce qui bloquait la connexion PDO. MariaDB 10.6.5 sur 3307 accepte la connexion sans problème.
3 Qu'est-ce qu'une SPA (Single Page Application) comme Angular ?
Réponse : B. Dans app.routes.ts, chaque route utilise loadComponent: () => import(...) — le routeur Angular change la vue sans recharger la page. Seules les requêtes API (ex. GET /api/produits) partent vers PHP. Le serveur ne génère jamais de HTML, il renvoie uniquement du JSON.
4 Quel fichier PHP est le point d'entrée unique de toutes les requêtes HTTP ?
Réponse : D. public/index.php fait require bootstrap.php puis $app->run(). Slim intercepte l'URL, cherche une route dans routes.php et appelle le bon contrôleur. Pourquoi un seul point d'entrée ? Pour centraliser la gestion des erreurs, des middlewares et du routage.
5 Quel fichier Angular contient l'URL du backend (http://localhost:8000) ?
Réponse : C. Dans src/environments/environment.ts : export const environment = { apiUrl: 'http://localhost:8000' }. Tous les services importent cet objet : private readonly base = environment.apiUrl (ex. auth.service.ts ligne 10). Pourquoi ? Changer l'URL de prod ne touche qu'un seul fichier.

B. Angular — composants, services et routing

6 D'où partent toutes les requêtes HTTP dans l'application Angular ?
Réponse : A. Ex. dans auth.service.ts ligne 24 : this.http.post(`${this.base}/login`, body, { withCredentials: true }). Le composant login.component.ts appelle simplement this.auth.login(email, mdp).subscribe(...) — il ne touche pas à HttpClient. Pourquoi ? Séparation des responsabilités : le composant gère l'UI, le service gère le réseau.
7 À quoi sert le fichier app.config.ts ?
Réponse : B. Dans app.config.ts : providers: [ provideZoneChangeDetection(...), provideRouter(routes), provideHttpClient(withFetch()) ]. C'est l'équivalent de l'ancien NgModule mais en mode standalone (Angular 17+). Sans provideHttpClient(), aucun service ne peut injecter HttpClient.
8 Qu'est-ce qu'un Signal Angular (signal()) ?
Réponse : D. Dans auth.service.ts lignes 12-16 : currentUser = signal<Utilisateur | null>(null) puis isLoggedIn = computed(() => this.currentUser() !== null). Quand currentUser.set(data) est appelé après le login, tous les templates qui lisent isLoggedIn() se mettent à jour instantanément — sans EventEmitter ni Subject.
9 Quelle est la différence entre signal() et computed() ?
Réponse : C. Dans artisan-shop.component.ts lignes 110-113 : filteredProduits = computed(() => { const f = this.activeFilter(); return f ? this.produits().filter(p => p.categorie === f) : this.produits(); }). Pourquoi computed() et pas un getter classique ? Parce qu'il est réactif : il se recalcule automatiquement quand activeFilter ou produits changent, sans appel explicite.
10 À quoi sert un guard Angular (authGuard, roleGuard) ?
Réponse : A. Dans auth.guard.ts : if (auth.isLoggedIn()) return true; return router.createUrlTree(['/login']). Dans app.routes.ts ligne 19 : { path: 'commande', canActivate: [authGuard] }. Pourquoi ? Empêcher un visiteur non connecté d'accéder à la page de commande côté client (la protection côté serveur existe aussi dans AuthMiddleware).
11 Qu'est-ce que le lazy loading dans app.routes.ts ?
Réponse : B. Dans app.routes.ts ligne 30 : loadComponent: () => import('./pages/artisan/dashboard/artisan-dashboard.component').then(m => m.ArtisanDashboardComponent). Pourquoi ? Un client ne chargera jamais le code admin. Sans lazy loading, tout le JavaScript (admin + artisan + client) serait téléchargé dès la 1ère visite, ralentissant inutilement la page.
12 Pourquoi utilise-t-on FormData au lieu de JSON pour les requêtes POST vers le backend ?
Réponse : D. Dans auth.service.ts lignes 21-24 : const body = new FormData(); body.append('email', email); body.append('mot_de_passe', mot_de_passe); this.http.post('.../login', body, { withCredentials: true }). Côté PHP, AuthController lit $_POST['email']. Si on envoyait du JSON brut (Content-Type: application/json), $_POST serait vide — il faudrait json_decode(file_get_contents('php://input')).

C. PHP/Slim 4 — middleware et routing

13 Dans Slim 4, dans quel ordre les middlewares s'exécutent-ils ?
Réponse : C. Dans bootstrap.php lignes 22-24 : $app->add(new CorsMiddleware()); $app->add(new RequestDataMiddleware()); $app->add(new SessionMiddleware()). SessionMiddleware est ajouté en dernier → il s'exécute en premier à la réception (LIFO). Pourquoi LIFO ? Slim empile les middlewares comme des couches d'oignon : chaque couche enveloppe la suivante.
14 Quel middleware PHP est responsable du lancement de session_start() ?
Réponse : A. Dans app/middleware/SessionMiddleware.php, la méthode process() appelle session_start() avant de transmettre la requête : $handler->handle($request). Pourquoi un middleware dédié ? Pour centraliser le démarrage de session — sans lui, $_SESSION n'existe pas et AuthMiddleware ne peut pas vérifier $_SESSION['user_id'].
15 À quoi sert CorsMiddleware.php dans ce projet ?
Réponse : B. Dans CorsMiddleware.php : private const ALLOWED_ORIGINS = ['http://localhost:4200', 'http://127.0.0.1:4200']. La méthode addCorsHeaders() ajoute Access-Control-Allow-Origin, Access-Control-Allow-Credentials: true et les méthodes autorisées. Pourquoi ? Le navigateur bloque par sécurité toute requête vers une origine différente (port différent = origine différente).
16 Qu'est-ce qu'une requête preflight OPTIONS en CORS ?
Réponse : D. Dans CorsMiddleware.php lignes 22-24 : if ($request->getMethod() === 'OPTIONS') { return $this->addCorsHeaders($response, $allowedOrigin)->withStatus(204); }. Pourquoi ? Avant un POST cross-origin, le navigateur envoie d'abord un OPTIONS pour demander l'autorisation. Si le serveur ne répond pas correctement, la vraie requête est annulée avant même d'être envoyée.
17 Dans le projet, qu'est-ce que le pattern Singleton utilisé dans Database.php ?
Réponse : C. Dans app/models/Database.php : if (self::$pdo === null) { ... self::$pdo = new PDO($dsn, $user, $password, [...]) } puis return self::$pdo. Pourquoi ? Ouvrir une connexion PDO coûte du temps. Avec le singleton, ProduitModel, CommandeModel et UtilisateurModel partagent tous la même connexion sans la recréer.
18 Que fait AuthMiddleware.php quand $_SESSION['user_id'] est absent ?
Réponse : A. Dans app/middleware/AuthMiddleware.php lignes 15-19 : if (empty($_SESSION['user_id'])) { return JsonResponder::write($response, 401, ['message' => 'Authentification requise']) }. Pourquoi JSON et pas une redirection ? Le backend est une API pure — il ne sait pas que le client est un navigateur. C'est Angular qui gère la redirection vers /login en interceptant le 401.
19 Quelle est la différence entre AuthMiddleware et RoleMiddleware ?
Réponse : B. AuthMiddleware vérifie empty($_SESSION['user_id']) → 401 si absent. RoleMiddleware vérifie $_SESSION['id_role'] === $requiredRole → 403 si mauvais rôle. Exemple : un artisan qui appelle /api/admin/artisans reçoit 401 s'il n'est pas connecté, ou 403 s'il est connecté mais pas admin.

D. Sécurité et gestion des rôles

20 Quels sont les trois rôles définis dans l'application ?
Réponse : C. Dans auth.service.ts lignes 14-16 : isAdmin = computed(() => this.currentUser()?.id_role === 1); isArtisan = computed(() => this.currentUser()?.id_role === 2); isClient = computed(() => this.currentUser()?.id_role === 3). Ces signals computed pilotent les menus de navigation et les sections conditionnelles du header en temps réel.
21 Pourquoi y a-t-il une double protection (guard Angular + middleware PHP) ?
Réponse : D. Le guard Angular (role.guard.ts) empêche l'affichage de la page admin dans le navigateur. Mais un utilisateur malin pourrait appeler directement GET /api/admin/artisans avec Postman ou curl. Le RoleMiddleware PHP refuse alors avec un 403. Conclusion : le guard = confort UI, le middleware = vraie sécurité.
22 Pourquoi utilise-t-on withCredentials: true dans toutes les requêtes Angular ?
Réponse : A. Dans auth.service.ts ligne 48 : this.http.get(`${this.base}/profile`, { withCredentials: true }). Idem dans cart.service.ts ligne 18. Pourquoi ? Par défaut, le navigateur ne joint pas les cookies aux requêtes cross-origin. Avec withCredentials: true, le cookie PHPSESSID est envoyé → PHP reconnaît la session → $_SESSION['user_id'] est disponible.
23 Quel guard Angular protège les routes /artisan/* ?
Réponse : B. Dans app.routes.ts ligne 27 : canActivate: [roleGuard(2)]. Dans role.guard.ts : export const roleGuard = (role: number): CanActivateFn => () => { ... if (user.id_role === role || user.id_role === 1) return true; return router.createUrlTree(['/']) }. Remarque : l'admin (id=1) peut tout faire — il passe même roleGuard(2).

E. Flux de données et cycle de vie

24 Dans quel hook de cycle de vie Angular charge-t-on les données depuis l'API ?
Réponse : C. Dans artisan-shop.component.ts ligne 122 : ngOnInit() { this.route.params.subscribe(p => { const id = +p['id']; this.artisanSvc.getById(id).subscribe(...) }) }. Pourquoi pas le constructeur ? Le constructeur est exécuté avant que Angular ait lié les paramètres de route et les inputs — ngOnInit() est garanti après cette initialisation.
25 Comment fonctionne le badge "nombre d'articles" dans le header du panier ?
Réponse : D. Dans cart.service.ts lignes 11-12 : lines = signal<CartLine[]>([]); count = computed(() => this.lines().reduce((s, l) => s + l.quantite, 0)). Le header lit cartService.count() dans son template. Quand cart.add() recharge les lignes avec this.lines.set(...), count se recalcule et le badge se met à jour sans code supplémentaire.
26 Quel fichier TypeScript définit les interfaces Produit, Commande, Artisan… ?
Réponse : A. Dans src/app/core/models/models.ts : export interface Produit { id_produit: number; nom: string; prix_ht: number; id_artisan: number; ... }. Pourquoi ces noms avec underscore ? Pour correspondre exactement aux colonnes MySQL. Si un champ PHP change de nom, TypeScript signale immédiatement une erreur partout où il est utilisé.
27 Quelles sont les valeurs possibles du statut d'une commande ?
Réponse : B. Dans models.ts : export type StatutCommande = 'en_attente' | 'en_preparation' | 'expediee' | 'livree' | 'annulee'. Ces valeurs correspondent aux ENUM dans la table commande de MariaDB. Côté UI, STATUT_LABELS les traduit en texte lisible (ex. en_attente → "En attente").

F. Configuration et outils

28 Pourquoi a-t-on dû ajouter "zone.js" dans angular.json ?
Réponse : D. Dans angular.json : "polyfills": ["zone.js"]. Zone.js intercepte toutes les opérations asynchrones (Promises, setTimeout, events) et notifie Angular pour déclencher la détection de changements. Sans lui → la page est blanche avec l'erreur console NG0908: Angular requires Zone.js to be loaded.
29 Quel fichier configure Tailwind CSS pour fonctionner avec Angular ?
Réponse : C. Dans postcss.config.js : module.exports = { plugins: { tailwindcss: {}, autoprefixer: {} } }. Pourquoi ? Le builder Angular (@angular/build:application) passe le SCSS par PostCSS. Sans ce fichier, les directives @tailwind base, @tailwind components dans styles.scss ne sont pas transformées et Tailwind ne génère aucune classe CSS.
30 À quoi sert BasePathResolver.php dans Slim 4 ?
Réponse : A. Dans bootstrap.php ligne 15 : $basePath = BasePathResolver::resolve($config). Avec php -S localhost:8000, SCRIPT_NAME = /index.php → base path vide, les routes sont directement /api/produits. Sous Apache dans /marketplace/public/, le base path serait /marketplace/public. BasePathResolver détecte ça automatiquement.
31 Quelle est la structure MVC du backend PHP ?
Réponse : B. Ex : ProductController::getAll() appelle ProduitModel::getAll() (Model) puis JsonResponder::write($response, 200, $data) (View JSON). Il n'y a aucun template HTML — Angular génère toute l'UI. models/ ne fait que du PDO pur, aucune logique métier.
32 Quel est le rôle des fichiers dans app/models/validators/ ?
Réponse : D. Ex : avant d'insérer un produit, ProductController appelle ProduitValidator::validate($data) qui vérifie que nom, prix_ht, stock sont présents et valides. Si non → exception ValidationException → réponse 422 avec les erreurs. Pourquoi séparer ? Le modèle ne fait que du SQL — la validation est une responsabilité distincte (SRP).
