Basé sur le projet ERA (module SRDA) réalisé par Neame Ramy en stage chez NSI IT — Angular 21 + NgCube (frontend) + Spring Boot + JCube + Hibernate + Liquibase + MapStruct + PostgreSQL (backend).

A. Architecture JCube & Spring Boot

1 Qu'impose JCube comme architecture dans le projet ERA ?
Réponse : B. JCube est le framework interne de NSI IT. Il impose 5 couches : EmployeeController (routes REST), EmployeeService (logique métier), EmployeeRepository (Spring Data JPA), EmployeeMapper (MapStruct), EmployeeEntity (Hibernate). Tout contrôleur étend AbstractCodeController, tout service étend AbstractCodeService. Cette convention garantit qu'un développeur NSI IT comprend n'importe quelle feature en moins d'une heure — contrairement au marketplace Slim où chaque développeur peut structurer librement ses contrôleurs.
2 Combien de fichiers la commande npm run plop génère-t-elle pour chaque nouvelle entité JCube ?
Réponse : C. npm run plop demande le nom de l'entité et génère 11 fichiers : EmployeeEntity.java, EmployeeDTO.java (record), EmployeeMapper.java (interface MapStruct), EmployeeService.java, EmployeeRepository.java, EmployeeController.java, EmployeeSearchCriteria.java, plus les fichiers de test et le changeset Liquibase. Dans le marketplace Slim, ces équivalents sont créés manuellement — environ 30 à 60 minutes contre 5 minutes avec Plop. Le risque d'oublier un fichier = quasi nul avec Plop, élevé sans.
3 Que font @RestController et @RequestMapping("/api/employees") combinés ?
Réponse : A. Dans EmployeeController.java : @RestController @RequestMapping("/api/employees") déclare que cette classe gère les requêtes vers /api/employees. Chaque méthode annotée @GetMapping("/{code}") ou @PostMapping("/search") devient un endpoint REST. Spring analyse ces annotations au démarrage et configure le routing automatiquement — contrairement au marketplace où toutes les routes sont déclarées explicitement dans routes.php : $app->get('/api/produits/{id}', [ProductController::class, 'getById']).
4 Pourquoi le projet ERA utilise POST /search pour les recherches plutôt que GET /search?param= ?
Réponse : D. Dans EmployeeController.java : @PostMapping("/search") public Page<EmployeeDTO> search(@RequestBody SearchCriteria criteria, Pageable pageable). Les URLs GET ont une limite (~2000 caractères) et exposent les critères dans les logs serveur. Le body JSON POST contient {"firstName": "Ramy", "buCode": "IT", "page": 0, "size": 20, "sort": ["lastName,asc"]} sans contrainte de taille. C'est une convention interne NSI IT adoptée dans tout le projet ERA et gérée automatiquement par LazyListManager côté NgCube.
5 Qu'est-ce que AbstractCodeController dans l'architecture JCube ?
Réponse : B. Dans ERA : public class EmployeeController extends AbstractCodeController<EmployeeDTO, EmployeeEntity, String>. AbstractCodeController est fourni par JCube et contient déjà les méthodes CRUD génériques (findAll, findById, create, update, delete). Les contrôleurs concrets n'ajoutent que la logique spécifique. L'équivalent dans le marketplace PHP : chaque ProductController.php et OrderController.php réimplémente son propre CRUD depuis zéro — plus de code, plus de risque d'incohérence entre contrôleurs.

B. ORM Hibernate / JPA

6 Que font @Entity et @Table(name = "employee") sur une classe Java ?
Réponse : C. Dans EmployeeEntity.java : @Entity @Table(name = "employee") public class EmployeeEntity. Hibernate lit ces annotations et sait que chaque instance correspond à une ligne dans la table SQL employee. L'annotation @Id @Column(name = "code") private String code mappe le champ Java à la colonne SQL. Dans le marketplace, ce mapping est absent — ProduitModel.php écrit directement SELECT * FROM produit WHERE id_artisan = ? sans aucune annotation. Hibernate génère ce SQL automatiquement depuis les annotations.
7 Que fait @ManyToOne(fetch = FetchType.LAZY) sur une relation Hibernate ?
Réponse : A. Dans EmployeeEntity.java : @ManyToOne(fetch = FetchType.LAZY) @JoinColumn(name = "bu_code") private BusinessUnitEntity businessUnit. Hibernate ne charge businessUnit que si on appelle employee.getBusinessUnit() dans le code. Sans LAZY, EAGER chargerait la BusinessUnit pour chaque employee même si on n'en a pas besoin — problème de performance. Dans le marketplace, une jointure SQL JOIN utilisateur u ON u.id = p.id_artisan charge toujours les données de l'artisan, qu'on en ait besoin ou non dans la réponse.
8 Qu'est-ce que le problème N+1 en ORM et comment Hibernate le résout-il ?
Réponse : B. Sans optimisation : findAll() fait 1 requête pour 50 employees, puis 50 requêtes pour charger la BusinessUnit de chacun = 51 requêtes SQL. Avec @EntityGraph(attributePaths = {"businessUnit"}), Hibernate génère un seul JOIN SQL : SELECT e.*, bu.* FROM employee e LEFT JOIN business_unit bu ON bu.code = e.bu_code. Dans le marketplace PHP, la solution équivalente est d'écrire le JOIN manuellement : SELECT p.*, u.nom FROM produit p JOIN utilisateur u ON u.id = p.id_artisan — mais sans framework ORM pour détecter le problème automatiquement.
9 Que fait l'annotation @Transactional sur une méthode de service Spring Boot ?
Réponse : D. Dans CommandeService.java : @Transactional public Commande creerCommande(CommandeDTO dto). Si la méthode se termine normalement, Spring fait commit(). Si une RuntimeException est levée (ex : stock insuffisant), Spring fait rollback() automatiquement — sans qu'on écrive une seule ligne de gestion de transaction. Dans le marketplace OrderController.php, la même logique nécessite $pdo->beginTransaction(), un bloc try/catch, $pdo->commit() et $pdo->rollBack() explicites — plus de code, plus de risque d'oublier le rollback.
10 Qu'est-ce que le Dirty Checking d'Hibernate ?
Réponse : A. Hibernate prend un "snapshot" de l'entité au moment du chargement (find()). À la fin de la transaction (flush()), il compare l'état actuel au snapshot et génère automatiquement un UPDATE pour les champs modifiés — même sans appel explicite à save(). Dans le marketplace PHP, un UPDATE doit être écrit explicitement : $stmt = $pdo->prepare('UPDATE produit SET prix_ht = ? WHERE id_produit = ?'); $stmt->execute([$prix, $id]). Oublier cet appel = aucune modification persistée, sans erreur.

C. Liquibase & gestion BDD

11 Quel est le rôle principal de Liquibase dans le projet ERA ?
Réponse : C. Liquibase maintient une table DATABASECHANGELOG dans PostgreSQL. Chaque changeset (ex : changelog-2024-001-add-employee-table.xml) a un id unique — Liquibase vérifie si cet id est déjà dans DATABASECHANGELOG. Si oui, il est ignoré. Si non, il est exécuté et enregistré. Résultat : impossible d'appliquer le même changeset deux fois, impossible d'oublier une migration. Dans le marketplace, les scripts SQL dans sql/Script SQL.sql sont appliqués manuellement via phpMyAdmin — aucune trace de ce qui a été exécuté où.
12 Dans quel format Plop génère-t-il les changesets Liquibase dans le projet ERA ?
Réponse : D. Liquibase supporte XML, YAML, JSON et SQL. Dans le projet ERA généré par Plop, les changesets sont en XML : <changeSet id="2024-001" author="plop"><createTable tableName="employee"><column name="code" type="VARCHAR(10)"><constraints primaryKey="true"/></column></createTable></changeSet>. L'avantage du XML : lisible, validable par schéma XSD, indépendant du SGBD (le même changeset fonctionne sur PostgreSQL et MySQL). Dans le marketplace, l'équivalent est sql/Script SQL.sql — non versionné, non tracé, spécifique à MariaDB.
13 Que se passe-t-il si on modifie le contenu d'un changeset Liquibase déjà appliqué en production ?
Réponse : B. Liquibase calcule un checksum MD5 de chaque changeset et le stocke dans DATABASECHANGELOG. Si le contenu change, le checksum change. Au prochain démarrage, Liquibase compare et lève ValidationFailedException: Validation Failed: 1 change sets check sum. C'est une protection contre la modification accidentelle de l'historique. La règle absolue dans ERA : un changeset appliqué ne se modifie jamais — on crée un nouveau changeset pour corriger, ex : changelog-2024-002-alter-employee-add-phone.xml.
14 Quel avantage a Liquibase sur les scripts SQL manuels pour déployer sur un nouveau serveur ?
Réponse : A. Sur un nouveau serveur (ex : staging), mvn spring-boot:run exécute tous les changesets non encore appliqués dans l'ordre. En 30 secondes, la base est au même niveau que la production — sans documentation, sans risque d'oubli. Lors du déploiement sur bacinfo.eci-liege.info pour le marketplace, il a fallu importer manuellement Script SQL.sql via phpMyAdmin, puis SeedDataClone.sql. Avec Liquibase, cette étape manuelle disparaît entièrement.
15 Que fait liquibase rollbackCount 1 dans le projet ERA ?
Réponse : C. liquibase rollbackCount 1 annule le dernier changeset — si c'était un createTable, il exécute un dropTable. Mais le rollback n'est pas automatique pour tous les changements : modifier des données ou supprimer une colonne nécessite de spécifier explicitement <rollback>ALTER TABLE employee ADD COLUMN phone VARCHAR(20)</rollback>. Dans le marketplace, l'équivalent est d'écrire manuellement ALTER TABLE produit DROP COLUMN nouveau_champ ou de restaurer un dump SQL — opération risquée et non automatisée.

D. MapStruct & DTOs

16 Pourquoi utilise-t-on un DTO plutôt que de retourner directement l'entité Hibernate en JSON ?
Réponse : B. Retourner directement EmployeeEntity avec @ManyToOne BusinessUnitEntity : Jackson tente de sérialiser la BusinessUnit, qui contient des Employee, qui contient des BusinessUnit... boucle infinie (StackOverflowError). De plus, EmployeeEntity peut exposer des champs internes (clés étrangères brutes, hash de mot de passe). Le EmployeeDTO est une projection propre : public record EmployeeDTO(String code, String firstName, String buLabel) {}. Dans le marketplace, ProduitModel.php retourne directement le tableau SQL — les clés étrangères (id_artisan) sont exposées par défaut dans la réponse JSON.
17 Qu'est-ce qu'un Java Record utilisé comme DTO dans le projet ERA ?
Réponse : D. Dans ERA : public record EmployeeDTO(String code, String firstName, String lastName, LocalDate birthDate, String buCode). Le compilateur génère automatiquement : un constructeur avec tous les paramètres, des accesseurs code(), firstName()..., equals(), hashCode() et toString(). Immuable par conception — impossible de modifier un champ après création. En PHP, l'équivalent depuis PHP 8.2 est readonly class EmployeeDTO { public function __construct(public readonly string $code, public readonly string $firstName) {} } — non utilisé dans le marketplace, qui retourne de simples tableaux PHP.
18 Que fait MapStruct à la compilation du projet Java ?
Réponse : A. L'interface EmployeeMapper annotée @Mapper(componentModel = "spring") déclare EmployeeDTO toDto(EmployeeEntity e). À la compilation (mvn compile), MapStruct génère EmployeeMapperImpl.java — une vraie classe Java : return new EmployeeDTO(e.getCode(), e.getFirstName(), e.getBusinessUnit().getLabel()). Aucune réflexion à l'exécution → performances maximales. Dans le marketplace, ce mapping est fait manuellement dans les contrôleurs PHP : return ['id' => $row['id_produit'], 'nom' => $row['nom_produit'], ...] — risque d'oublier un champ si la structure change.
19 Que se passe-t-il si un champ du DTO n'a pas de correspondance dans l'entité avec MapStruct ?
Réponse : C. Avec @Mapper(unmappedTargetPolicy = ReportingPolicy.ERROR), si EmployeeDTO a un champ buLabel mais que EmployeeEntity n'a pas ce champ directement (il vient de businessUnit.label), MapStruct génère une erreur de compilation. Cela force à déclarer explicitement @Mapping(source = "businessUnit.label", target = "buLabel"). Avantage : les erreurs de mapping sont détectées à la compilation, pas en production. En PHP, un champ mal mappé dans le tableau de retour du contrôleur ne provoque aucune erreur — le frontend Angular reçoit simplement undefined.
20 Quelle est la différence entre toDto() et toEntity() dans un Mapper MapStruct ?
Réponse : B. Dans EmployeeMapper.java : EmployeeDTO toDto(EmployeeEntity e) est utilisé dans EmployeeService.findById() avant de retourner la réponse HTTP. EmployeeEntity toEntity(EmployeeDTO dto) est utilisé dans EmployeeService.create() avant d'appeler repository.save(). Cette séparation garantit que l'API ne reçoit jamais directement une entité brute et que la BDD ne reçoit jamais un DTO non validé. Dans le marketplace, les contrôleurs PHP passent directement les données $_POST au modèle sans couche de conversion dédiée.

E. Sécurité & JWT

21 Quelle est la différence fondamentale entre JWT (NSI IT) et session PHP (marketplace) ?
Réponse : D. JWT (NSI IT) : le token eyJhbGci... contient {"userId": 42, "role": "RESPONSABLE", "exp": 1234567890} signé — le serveur vérifie la signature sans chercher en base ou en mémoire. Scalable : n'importe quel serveur du cluster peut valider le token. Session PHP (marketplace) : $_SESSION['user_id'] = 42 est stocké dans un fichier sur le serveur, le cookie PHPSESSID pointe vers ce fichier. Sur 3 serveurs en parallèle (load balancer), un utilisateur connecté sur le serveur 1 sera "déconnecté" sur le serveur 2 — problème de session partagée.
22 Que fait @PreAuthorize("hasRole('ROLE_RESPONSABLE')") dans Spring Security ?
Réponse : B. @PreAuthorize("hasRole('ROLE_RESPONSABLE')") sur deleteEmployee() dans EmployeeService.java est évalué AVANT l'exécution — même si un AdminService interne appelle employeeService.deleteEmployee(). Spring Security utilise AOP pour intercepter l'appel. Dans le marketplace, RoleMiddleware.php ne protège que les routes HTTP — si AdminController.php appelle une méthode de ProductController.php directement en PHP, aucune vérification de rôle n'est appliquée. La protection @PreAuthorize est plus profonde et plus sûre.
23 Comment Angular envoie-t-il le token JWT au backend Spring Boot dans le projet ERA ?
Réponse : A. Dans le projet ERA, un AuthInterceptor Angular intercepte toutes les requêtes HttpClient et ajoute headers: { Authorization: 'Bearer ' + localStorage.getItem('token') }. Le backend Spring Security lit ce header et valide la signature JWT avec la clé secrète définie dans application.properties. Contrairement au marketplace où Angular utilise withCredentials: true pour envoyer le cookie PHPSESSID automatiquement — mécanisme navigateur qui pose des problèmes CORS avec Access-Control-Allow-Credentials: true.
24 Quelles informations peut-on encoder dans le payload d'un token JWT ?
Réponse : C. Un JWT a 3 parties : header (algorithme HMAC/RSA), payload (claims), signature. Dans ERA : {"sub": "EMP001", "roles": ["ROLE_RESPONSABLE"], "bu": "IT", "exp": 1234567890, "iat": 1234560000}. Ces claims permettent au serveur de répondre sans requête BDD : userId, rôle, unité sont disponibles directement. Attention : le payload JWT n'est pas chiffré (juste base64) — ne jamais y mettre de mot de passe ou données sensibles. Dans le marketplace, $_SESSION['user_id'] ne contient que l'ID — chaque requête nécessite un SELECT * FROM utilisateur WHERE id_utilisateur = ?.
25 Que vérifie Spring Security lors de la réception d'un token JWT sur chaque requête ?
Réponse : D. Spring Security JWT filter : 1) extrait le token du header Authorization: Bearer ..., 2) vérifie la signature avec la clé secrète (jwt.secret dans application.properties) — si le token a été falsifié, la signature ne correspond plus → 401, 3) vérifie que exp > timestamp actuel — token expiré → 401, 4) extrait les claims roles et construit le SecurityContext pour les @PreAuthorize. Aucune requête BDD nécessaire — c'est l'avantage stateless JWT. Dans le marketplace : if (empty($_SESSION['user_id'])) dans AuthMiddleware.php — vérifie juste que le fichier de session PHP existe côté serveur.

F. Angular NgCube & communication

26 Qu'est-ce que NgCube dans le projet ERA et quel est son lien avec JCube ?
Réponse : A. NgCube est la bibliothèque Angular interne de NSI IT, complémentaire à JCube côté backend. Elle fournit des composants réutilisables (tableaux, formulaires, filtres paginés) et des managers comme LazyListManager qui gère automatiquement la pagination, le tri et la recherche via POST /search. Dans le marketplace Angular, ces fonctionnalités sont implémentées manuellement — ex : filteredProduits = computed(() => { const f = this.activeFilter(); return f ? this.produits().filter(p => p.categorie === f) : this.produits(); }) dans artisan-shop.component.ts.
27 Que fait LazyListManager de NgCube dans le projet ERA ?
Réponse : B. LazyListManager encapsule le cycle pagination/tri/filtre : quand l'utilisateur change de page ou trie une colonne, manager.load() envoie automatiquement POST /api/employees/search avec {"page": 1, "size": 20, "sort": "lastName,asc", ...criteria}. Le Signal manager.content() reçoit la réponse Page<EmployeeDTO> de Spring. Dans le marketplace, ce mécanisme est absent — ProductService.getAll() charge tous les produits sans pagination, et le filtrage se fait côté client dans artisan-shop.component.ts avec un Signal computed().
28 Pourquoi les interfaces TypeScript d'ERA utilisent le camelCase et le marketplace le snake_case ?
Réponse : C. Jackson (Spring Boot) sérialise EmployeeDTO.firstName → "firstName" en JSON par défaut. L'interface TypeScript reflète ce JSON : interface EmployeeDTO { code: string; firstName: string; buLabel: string; }. Dans le marketplace, PHP retourne les colonnes SQL telles quelles — id_produit, nom_produit, prix_ht — et les interfaces TypeScript dans models.ts reflètent ces noms snake_case : interface Produit { id_produit: number; nom_produit: string; prix_ht: number; }. Les deux sont valides mais les conventions divergent selon le langage backend.
29 Quelle est la différence entre withCredentials: true (marketplace) et un intercepteur JWT (ERA) dans Angular ?
Réponse : D. Marketplace (auth.service.ts) : this.http.post('/auth/login', formData, { withCredentials: true }) — le navigateur envoie automatiquement le cookie PHPSESSID sur chaque requête vers le même domaine. Côté PHP, CorsMiddleware.php doit répondre Access-Control-Allow-Credentials: true et Access-Control-Allow-Origin ne peut pas être *. ERA : un HttpInterceptor lit le token depuis localStorage.getItem('token') et ajoute headers.set('Authorization', 'Bearer ' + token) — plus explicite, indépendant du mécanisme cookie du navigateur, sans contrainte CORS spéciale.
30 Comment Angular détecte-t-il qu'un token JWT est expiré ou invalide dans le projet ERA ?
Réponse : A. Dans ERA, quand le token expire, la prochaine requête HTTP reçoit une réponse 401 Unauthorized de Spring Security. Un ErrorInterceptor Angular intercepte les réponses avec status === 401, efface le token du localStorage et redirige vers /login. Optionnellement, l'intercepteur peut aussi lire le claim exp du JWT localement (base64 decode du payload) pour rediriger proactivement avant que la requête parte. Dans le marketplace, authGuard dans auth.guard.ts vérifie auth.isLoggedIn() (Signal) avant la navigation — si le cookie expire, la prochaine requête reçoit un 401 de AuthMiddleware.php.
